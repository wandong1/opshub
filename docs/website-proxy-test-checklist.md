# Web站点代理功能测试清单

## 测试环境准备

### 1. 启动服务
```bash
# 启动后端服务
go run cmd/server/server.go server -c config/config.yaml

# 启动前端（另一个终端）
cd web && npm run dev
```

### 2. 部署 Agent
```bash
# 在测试主机上部署 Agent
# 方式1：通过平台 SSH 部署
# 方式2：手动安装
cd agent
./build.sh
# 将生成的安装包复制到目标主机并安装
```

## 功能测试

### 测试1：外部站点直接访问
**目的**：验证外部站点可以直接打开

**步骤**：
1. 登录平台
2. 进入"资产管理" -> "Web站点管理"
3. 点击"新增站点"
4. 填写表单：
   - 站点名称：百度
   - 站点类型：外部站点
   - 站点URL：https://www.baidu.com
   - 状态：启用
5. 点击"确定"
6. 在站点列表中找到刚创建的站点
7. 点击"访问"按钮

**预期结果**：
- ✅ 新标签页打开百度首页
- ✅ 无需 Agent，直接访问

---

### 测试2：内部站点通过 Agent 代理访问
**目的**：验证内部站点可以通过 Agent 代理访问

**前置条件**：
- 至少有一台 Agent 在线
- 内网有一个可访问的 Web 服务（如内网 Nginx、内网应用等）

**步骤**：
1. 进入"Web站点管理"
2. 点击"新增站点"
3. 填写表单：
   - 站点名称：内网应用
   - 站点类型：内部站点
   - 站点URL：http://192.168.1.100:8080（替换为实际内网地址）
   - 绑定Agent主机：选择一台在线的 Agent
   - 状态：启用
4. 点击"确定"
5. 在站点列表中找到刚创建的站点
6. 确认"Agent状态"列显示"Agent在线"
7. 点击"访问"按钮

**预期结果**：
- ✅ 新标签页打开代理 URL（/api/v1/websites/:id/proxy）
- ✅ 显示内网应用的完整页面内容
- ✅ 页面中的静态资源（CSS、JS、图片）正常加载
- ✅ 响应体大小不受 4KB 限制

---

### 测试3：Agent 离线时的错误提示
**目的**：验证 Agent 离线时的错误处理

**步骤**：
1. 停止测试主机上的 Agent 服务
   ```bash
   systemctl stop srehub-agent
   ```
2. 等待 30 秒（心跳超时）
3. 刷新"Web站点管理"页面
4. 确认内部站点的"Agent状态"列显示"Agent离线"
5. 点击"访问"按钮

**预期结果**：
- ✅ 显示错误提示："Agent主机离线，无法访问"
- ✅ HTTP 状态码：503
- ✅ 不会尝试访问

---

### 测试4：多 Agent 主机的故障转移
**目的**：验证多 Agent 场景下的自动切换

**前置条件**：
- 至少有两台 Agent 在线

**步骤**：
1. 编辑一个内部站点
2. 在"绑定Agent主机"中选择两台 Agent
3. 保存
4. 点击"访问"按钮，记录使用的 Agent（通过后端日志）
5. 停止第一台 Agent
6. 再次点击"访问"按钮

**预期结果**：
- ✅ 第一次访问使用第一台 Agent
- ✅ 第一台 Agent 离线后，自动使用第二台 Agent
- ✅ 访问成功，无需用户干预

---

### 测试5：拨测功能不受影响
**目的**：验证 Web 站点代理功能不影响现有拨测功能

**步骤**：
1. 进入"巡检管理" -> "拨测配置"
2. 创建一个 HTTP 拨测配置：
   - 拨测名称：百度拨测
   - 拨测类型：HTTP
   - 目标URL：https://www.baidu.com
   - 执行模式：Agent
   - 绑定Agent：选择一台在线的 Agent
3. 保存并执行拨测
4. 查看拨测结果

**预期结果**：
- ✅ 拨测正常执行
- ✅ 返回 HTTP 状态码、响应时间等指标
- ✅ 响应体限制为 4KB（拨测场景的默认行为）
- ✅ 性能指标完整（DNS、连接、TLS、TTFB 等）

---

### 测试6：大型网页内容代理
**目的**：验证响应体大小限制优化是否生效

**步骤**：
1. 创建一个内部站点，指向一个内容较大的网页（如带图片的管理后台）
2. 通过 Agent 代理访问
3. 打开浏览器开发者工具，查看网络请求
4. 检查响应体大小

**预期结果**：
- ✅ 完整的网页内容（不截断）
- ✅ 响应体大小可超过 4KB
- ✅ 最大支持 10MB（防止内存溢出）

---

### 测试7：HTTP 方法支持
**目的**：验证支持各种 HTTP 方法

**步骤**：
1. 创建一个内部站点，指向一个 RESTful API
2. 使用 Postman 或 curl 测试不同的 HTTP 方法：
   ```bash
   # GET
   curl http://localhost:9876/api/v1/websites/1/proxy/api/users

   # POST
   curl -X POST http://localhost:9876/api/v1/websites/1/proxy/api/users \
     -H "Content-Type: application/json" \
     -d '{"name":"test"}'

   # PUT
   curl -X PUT http://localhost:9876/api/v1/websites/1/proxy/api/users/1 \
     -H "Content-Type: application/json" \
     -d '{"name":"updated"}'

   # DELETE
   curl -X DELETE http://localhost:9876/api/v1/websites/1/proxy/api/users/1
   ```

**预期结果**：
- ✅ 所有 HTTP 方法正常工作
- ✅ 请求头正确转发
- ✅ 请求体正确转发
- ✅ 响应头正确返回

---

### 测试8：请求头和响应头处理
**目的**：验证请求头和响应头的正确处理

**步骤**：
1. 使用 curl 发送带自定义请求头的请求：
   ```bash
   curl http://localhost:9876/api/v1/websites/1/proxy/api/test \
     -H "X-Custom-Header: test-value" \
     -H "Authorization: Bearer token123"
   ```
2. 检查目标服务器是否收到这些请求头
3. 检查响应头是否正确返回

**预期结果**：
- ✅ 自定义请求头正确转发（除了 Host、Connection、X-Forwarded-* 等）
- ✅ 响应头正确返回
- ✅ Content-Type、Content-Length 等关键头正确处理

---

### 测试9：查询参数和路径参数
**目的**：验证 URL 参数的正确处理

**步骤**：
1. 访问带查询参数的 URL：
   ```bash

   ```
2. 访问带路径参数的 URL：
   ```bash
   curl "http://localhost:9876/api/v1/websites/1/proxy/api/users/123/profile"
   ```

**预期结果**：
- ✅ 查询参数正确转发
- ✅ 路径参数正确拼接
- ✅ URL 编码正确处理

---

### 测试10：超时处理
**目的**：验证超时机制

**步骤**：
1. 创建一个内部站点，指向一个响应很慢的服务（或使用 sleep 模拟）
2. 通过 Agent 代理访问
3. 等待 30 秒（默认超时时间）

**预期结果**：
- ✅ 30 秒后返回超时错误
- ✅ 错误信息清晰："代理请求失败: request failed: context deadline exceeded"
- ✅ 不会无限等待

## 已知问题和改进建议

### 已知问题
1. **响应体大小限制**：虽然已优化到 10MB，但对于某些场景可能仍不够
2. **SSRF 防护**：当前未实现，存在安全风险
3. **缓存机制**：未实现响应缓存，每次都需要通过 Agent 请求

### 改进建议
1. **添加 SSRF 防护**：
   - 实现 IP 白名单/黑名单
   - 禁止访问内网敏感端口（22、3306、6379 等）
   - 仅允许管理员创建内部站点

2. **实现响应缓存**：
   - 对静态资源（CSS、JS、图片）进行缓存
   - 减少 Agent 负载
   - 提升访问速度

3. **添加访问日志**：
   - 记录每次代理访问（用户、站点、时间、状态）
   - 用于审计和分析

4. **支持 WebSocket 代理**：
   - 当前仅支持 HTTP/HTTPS
   - 未来可扩展支持 WebSocket

5. **优化错误提示**：
   - 前端显示更友好的错误信息
   - 区分不同的错误类型（Agent 离线、超时、目标不可达等）
