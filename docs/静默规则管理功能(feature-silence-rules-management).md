# 屏蔽规则管理功能优化说明

## 📋 优化内容

### 问题背景

**原有问题**：
1. "已屏蔽告警"列表显示的是告警事件（`alert_events` 表），相同三元组（告警等级 + 规则名称 + 标签）的告警会出现多条重复记录
2. 无法直接编辑屏蔽时间，需要先取消屏蔽再重新屏蔽
3. 用户体验不佳，管理屏蔽规则不够直观

**优化方案**：
1. 将"已屏蔽告警"改为"屏蔽规则管理"
2. 直接显示屏蔽规则列表（`alert_silence_rules` 表）
3. 支持直接编辑屏蔽规则的时间、类型、原因等信息
4. 支持启用/禁用屏蔽规则

---

## 🎯 核心改进

### 1. 数据源变更

#### 修改前
- 数据来源：`alert_events` 表（告警事件）
- 查询条件：`silenced = true`
- 问题：同一个屏蔽规则匹配的多条告警都会显示

#### 修改后
- 数据来源：`alert_silence_rules` 表（屏蔽规则）
- 查询条件：所有屏蔽规则
- 优势：每个三元组只显示一条记录，清晰明了

### 2. 功能对比

| 功能 | 修改前 | 修改后 |
|------|--------|--------|
| 列表内容 | 告警事件列表 | 屏蔽规则列表 |
| 数据去重 | ❌ 重复显示 | ✅ 自动去重 |
| 编辑屏蔽时间 | ❌ 不支持 | ✅ 支持 |
| 启用/禁用 | ❌ 不支持 | ✅ 支持 |
| 批量操作 | 取消屏蔽 | 删除规则 |
| 告警状态显示 | 告警中/已恢复 | 启用/禁用 |

## 🔧 功能详解

### 1. 查询屏蔽规则

**API**：`GET /api/v1/alert/silence-rules`

**参数**：
```typescript
{
  page: number
  pageSize: number
}
```

**响应**：
```json
{
  "code": 0,
  "data": [
    {
      "id": 1,
      "severity": "critical",
      "ruleName": "测试持续触发",
      "labels": "{\"__name__\":\"srehub_inspect_check_assertion_result\",\"business_group\":\"专门用于测试的业务组\"}",
      "type": "fixed",
      "duration": "2h",
      "silenceUntil": "2026-04-08T17:37:00Z",
      "reason": "固定时长屏蔽",
      "enabled": true,
      "createdAt": "2026-04-08T15:37:00Z"
    }
  ],
  "total": 1
}
```

### 2. 编辑屏蔽规则

**功能**：
- 修改屏蔽类型（固定时长 ↔ 周期性）
- 修改屏蔽时长（1h/2h/6h/12h/24h/168h）
- 修改周期性时间段（星期几 + 时间范围）
- 修改屏蔽原因

**API**：`PUT /api/v1/alert/silence-rules/:id`

**参数**：
```typescript
{
  type: 'fixed' | 'periodic'
  duration?: string  // 固定时长时必填
  timeRanges?: string  // 周期性时必填，JSON 格式
  reason?: string
}
```

**示例**：

固定时长：
```json
{
  "type": "fixed",
  "duration": "6h",
  "reason": "临时维护"
}
```

周期性：
```json
{
  "type": "periodic",
  "timeRanges": "[{\"weekdays\":[1,2,3,4,5],\"start\":\"09:00\",\"end\":\"18:00\"}]",
  "reason": "工作时间屏蔽"
}
```

### 3. 启用/禁用屏蔽规则

**功能**：
- 禁用后，屏蔽规则不再生效，但不会删除
- 启用后，屏蔽规则恢复生效

**API**：`PUT /api/v1/alert/silence-rules/:id/toggle`

**效果**：
- 禁用：新产生的告警不会被屏蔽
- 启用：新产生的告警会被屏蔽

### 4. 删除屏蔽规则

**功能**：
- 永久删除屏蔽规则
- 已屏蔽的告警事件不会自动恢复（需要手动取消屏蔽）

**API**：`DELETE /api/v1/alert/silence-rules/:id`

**注意**：
- 删除后无法恢复
- 建议先禁用，确认无问题后再删除

## 🔄 业务流程

### 创建屏蔽规则

```
用户在实时告警页面点击"屏蔽"
    ↓
选择屏蔽类型和时长
    ↓
创建屏蔽规则（alert_silence_rules 表）
    ↓
更新告警事件的屏蔽状态（alert_events 表）
    ↓
新产生的告警自动匹配屏蔽规则
```

### 编辑屏蔽规则

```
用户在屏蔽规则管理页面点击"编辑"
    ↓
修改屏蔽类型、时长、时间段、原因
    ↓
更新屏蔽规则（alert_silence_rules 表）
    ↓
发布 Redis Pub/Sub 事件，通知评估引擎重载规则
    ↓
新产生的告警使用新的屏蔽规则
```

### 禁用屏蔽规则

```
用户在屏蔽规则管理页面点击"禁用"
    ↓
设置 enabled = false
    ↓
发布 Redis Pub/Sub 事件
    ↓
新产生的告警不再匹配该规则
```

### 删除屏蔽规则

```
用户在屏蔽规则管理页面点击"删除"
    ↓
删除屏蔽规则（alert_silence_rules 表）
    ↓
发布 Redis Pub/Sub 事件
    ↓
新产生的告警不再匹配该规则
```

## 📊 数据模型

### alert_silence_rules 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | uint | 主键 |
| severity | string | 告警等级 |
| rule_name | string | 规则名称 |
| labels | string | 标签（JSON） |
| type | string | 屏蔽类型（fixed/periodic） |
| duration | string | 固定时长（如 "2h"） |
| silence_until | timestamp | 屏蔽截止时间 |
| time_ranges | string | 周期性时间段（JSON） |
| reason | string | 屏蔽原因 |
| created_by | uint | 创建人 ID |
| enabled | bool | 是否启用 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

### 三元组唯一性

屏蔽规则按以下三元组匹配告警：
1. **severity**（告警等级）
2. **rule_name**（规则名称）
3. **labels**（标签）

相同三元组的屏蔽规则在数据库中只有一条记录。

## 🎯 使用场景

### 场景 1：临时维护

**需求**：服务器维护期间，屏蔽所有相关告警 2 小时

**操作**：
1. 在实时告警页面，选择相关告警
2. 点击"批量屏蔽"
3. 选择"固定时长" → "2小时"
4. 输入原因："服务器维护"
5. 确认屏蔽

**效果**：
- 2 小时内，相同告警不会推送通知
- 2 小时后，屏蔽自动失效

### 场景 2：工作时间屏蔽

**需求**：工作时间（周一至周五 09:00-18:00）屏蔽某些低优先级告警

**操作**：
1. 在实时告警页面，选择相关告警
2. 点击"屏蔽"
3. 选择"周期性"
4. 勾选"周一"至"周五"
5. 设置时间段："09:00" 至 "18:00"
6. 输入原因："工作时间屏蔽"
7. 确认屏蔽

**效果**：
- 每周一至周五 09:00-18:00，告警被屏蔽
- 其他时间，告警正常推送

### 场景 3：延长屏蔽时间

**需求**：原本屏蔽 2 小时，现在需要延长到 6 小时

**操作**：
1. 打开"屏蔽规则管理"
2. 找到对应的屏蔽规则
3. 点击"编辑"
4. 修改屏蔽时长："2h" → "6h"
5. 确认修改

**效果**：
- 屏蔽时间自动延长
- 无需重新创建屏蔽规则

### 场景 4：临时禁用屏蔽规则

**需求**：临时需要接收告警，但不想删除屏蔽规则

**操作**：
1. 打开"屏蔽规则管理"
2. 找到对应的屏蔽规则
3. 点击"禁用"

**效果**：
- 告警恢复正常推送
- 屏蔽规则保留，可随时启用

## ⚠️ 注意事项

### 1. 屏蔽规则的生效时机

- **新告警**：创建时自动检查屏蔽规则
- **已存在的告警**：不会自动更新屏蔽状态
- **建议**：修改屏蔽规则后，等待告警恢复再次触发

### 2. 屏蔽规则的优先级

- 固定时长和周期性屏蔽规则同时存在时，都会生效
- 只要匹配任一规则，告警就会被屏蔽

### 3. 删除屏蔽规则的影响

- 删除屏蔽规则后，新产生的告警不再被屏蔽
- 已屏蔽的告警事件不会自动恢复
- 如需恢复，需要在"已屏蔽告警"列表中手动取消屏蔽

### 4. 标签匹配规则

- 屏蔽规则的标签是**子集匹配**
- 告警的标签必须包含屏蔽规则的所有标签
- 例如：
  - 屏蔽规则标签：`{job: "prometheus"}`
  - 告警标签：`{job: "prometheus", instance: "localhost:9090"}` ✅ 匹配
  - 告警标签：`{job: "node_exporter"}` ❌ 不匹配

## 🔍 故障排查

### 问题 1：屏蔽规则不生效

**可能原因**：
1. 屏蔽规则被禁用
2. 标签不匹配
3. 固定时长已过期
4. 周期性时间段不在生效范围内

**排查步骤**：
1. 检查屏蔽规则状态是否为"启用"
2. 对比告警标签和屏蔽规则标签
3. 检查固定时长的截止时间
4. 检查周期性的星期几和时间段

### 问题 2：修改屏蔽规则后不生效

**可能原因**：
1. Redis 缓存未更新
2. 评估引擎未重载规则

**排查步骤**：
1. 检查后端日志，确认是否发布了 Pub/Sub 事件
2. 重启评估引擎（重启服务）
3. 等待告警恢复后再次触发

### 问题 3：删除屏蔽规则后告警仍显示"屏蔽中"

**原因**：
- 删除屏蔽规则只影响新产生的告警
- 已存在的告警事件的屏蔽状态不会自动更新

**解决方案**：
- 等待告警恢复后再次触发
- 或者手动取消屏蔽（在实时告警列表中操作）

## 📈 性能优化

### 1. 缓存机制

- 屏蔽规则从 Redis 缓存读取
- 避免每次评估都查询数据库
- 缓存自动订阅 Pub/Sub 事件，实时更新

### 2. 前端过滤

- 屏蔽规则列表在前端进行过滤
- 减少后端查询压力
- 提升用户体验

### 3. 批量操作

- 支持批量删除屏蔽规则
- 使用 `Promise.all` 并发执行
- 提升操作效率
