# 告警巡检功能 - 实施指南

## 📋 功能概述

告警巡检功能允许系统定期自动巡查符合订阅规则的告警，并生成美观的巡检报告推送给用户。这对于及时发现和处理长期未解决的告警非常有用。

---

## ✅ 实施步骤

### 1. 数据库迁移

巡检功能的表结构已经添加到 `migrations/init.sql` 文件中，包括：
- `alert_subscription_patrol` - 巡检配置表
- `alert_patrol_report` - 巡检报告表
- 相关权限配置

**执行方式：**
- 如果是新部署：直接运行 `init.sql` 即可
- 如果是已有系统：需要单独执行巡检相关的 SQL（从 `-- 19. 告警巡检表` 开始的部分）

**或者使用自动迁移：**
```bash
# 启动服务时会自动创建表结构
go run main.go server --config config/config.yaml
```

### 2. 启动服务

```bash
go run main.go server --config config/config.yaml
```

服务启动后会看到以下日志：

## 🎯 使用指南

### 1. 配置巡检

点击订阅列表中的"巡检配置"按钮，打开配置弹窗：

#### 基础配置
- **启用巡检**：开启/关闭巡检功能
- **推送模式**：
  - `总是推送`：无论有无告警都推送报告
  - `仅有告警时推送`：只在有活跃告警时推送

#### 巡检时间配置
- **巡检模式**：
  - `按间隔巡检`：每隔N秒执行一次（如每小时=3600秒）
  - `固定时间巡检`：在指定时间点执行（如 09:00、18:00）

#### 巡检范围配置
- **时间范围**：
  - `0`：巡检所有未恢复的告警
  - `>0`：只巡检最近N秒内的告警（如 86400=最近24小时）
- **最大告警数**：单次报告最多包含的告警数量（默认100）
- **包含已恢复**：是否在报告中统计已恢复的告警

#### 报告样式配置
- **报告样式**：
  - `详细报告`：显示每条告警的完整信息（最多20条）
  - `摘要报告`：只显示统计数据和Top告警（最多10条）
- **分组维度**：
  - `按告警级别分组`：critical、warning、info
  - `按规则名称分组`：相同规则的告警合并统计
  - `按业务分组`：按 asset_group 标签分组

#### 推送配置
- **推送通道**：留空则使用订阅的通道
- **接收用户**：留空则使用订阅的用户

### 2. 手动执行巡检

在配置弹窗中点击"手动执行巡检"按钮，可以立即触发一次巡检任务，用于测试配置是否正确。

### 3. 查看巡检报告

在配置弹窗中点击"查看巡检报告"按钮，可以查看历史巡检报告列表，包括：
- 巡检时间
- 活跃告警数量
- 各级别告警统计
- 是否已推送
- 推送时间

## 🔧 常见使用场景

### 场景1：每日早报
**需求**：每天早上9点推送昨日未解决告警汇总

**配置：**
- 巡检模式：`固定时间巡检`
- 巡检时间点：`["09:00"]`
- 时间范围：`86400`（24小时）
- 推送模式：`总是推送`
- 报告样式：`详细报告`

### 场景2：值班交接
**需求**：每4小时推送当前活跃告警清单

**配置：**
- 巡检模式：`按间隔巡检`
- 巡检间隔：`14400`（4小时）
- 时间范围：`0`（所有未恢复）
- 推送模式：`仅有告警时推送`
- 报告样式：`详细报告`

### 场景3：高危告警监控
**需求**：每小时巡检严重级别告警

**配置：**
- 巡检模式：`按间隔巡检`
- 巡检间隔：`3600`（1小时）
- 时间范围：`0`（所有未恢复）
- 推送模式：`仅有告警时推送`
- 报告样式：`摘要报告`
- 订阅规则中只选择 `critical` 级别

### 场景4：周报
**需求**：每周一早上推送上周告警统计

**配置：**
- 巡检模式：`固定时间巡检`
- 巡检时间点：`["09:00"]`（需要配合定时任务在周一执行）
- 时间范围：`604800`（7天）
- 包含已恢复：`开启`
- 推送模式：`总是推送`
- 报告样式：`摘要报告`

## 🎨 技术特性

### 1. 智能过滤
- 完全遵循订阅规则的过滤逻辑
- 支持时间范围、告警级别、数据源、标签匹配器等多维度过滤
- **不经过**去重、分组、抑制等治理规则（因为是汇总报告）

### 2. 灵活调度
- 支持间隔模式和固定时间模式
- 自动计算下次巡检时间
- 30秒检查一次，精确执行

### 3. 美观报告
- 使用 emoji 和分隔线美化报告
- 支持详细报告和摘要报告两种样式
- 自动格式化持续时长（如"2天3小时"）

### 4. 多通道推送
- 支持飞书、钉钉、企业微信、短信、邮件等
- 可独立配置巡检报告的推送通道和接收用户

## ⚠️ 注意事项

### 1. 巡检间隔建议
- 最小间隔：60秒（1分钟）
- 推荐间隔：3600秒（1小时）或更长
- 过短的间隔可能导致频繁推送，影响用户体验

### 2. 时间范围设置
- `0`：巡检所有未恢复的告警（推荐用于日常巡检）
- `>0`：只巡检最近N秒内的告警（推荐用于周报、月报）

### 3. 报告样式选择
- **详细报告**：适合告警数量较少（<20条）的场景
- **摘要报告**：适合告警数量较多的场景，避免报告过长

### 4. 推送模式选择
- **总是推送**：适合需要定期确认系统状态的场景（如早报）
- **仅有告警时推送**：适合只关注异常情况的场景（如高危告警监控）

### 5. 性能考虑
- 巡检服务每30秒检查一次，不会对数据库造成太大压力
- 单次巡检会查询订阅关联的所有告警，建议合理设置 `max_alerts_per_report`

## 🐛 故障排查

### 问题1：巡检没有执行
**检查：**
1. 巡检配置是否启用（`enabled = true`）
2. 查看服务日志是否有"告警巡检服务已启动"
3. 检查 `next_patrol_at` 字段是否正确

### 问题2：没有收到巡检报告
**检查：**
1. 推送模式是否为"仅有告警时推送"，但实际没有告警
2. 订阅的通道是否启用
3. 订阅的用户是否配置正确
4. 查看 `alert_patrol_report` 表中的 `sent` 和 `send_error` 字段

### 问题3：巡检报告中没有告警
**检查：**
1. 订阅规则的过滤条件是否过于严格
2. 时间范围设置是否合理
3. 手动执行巡检，查看是否有告警

### 问题4：巡检报告格式不美观
**检查：**
1. 通道类型是否支持 Markdown 格式（飞书、钉钉支持，短信不支持）
2. 尝试切换报告样式（详细/摘要）

## 📚 API 文档

### 1. 获取巡检配置
```
GET /api/v1/alert/patrol/:subscriptionId
```

**响应示例：**
```json
{
  "code": 0,
  "data": {
    "id": 1,
    "subscriptionId": 1,
    "enabled": true,
    "patrolMode": "interval",
    "patrolInterval": 3600,
    "patrolTimes": "[]",
    "includeResolved": false,
    "timeRange": 0,
    "maxAlertsPerReport": 100,
    "sendMode": "always",
    "channelIds": "[]",
    "userIds": "[]",
    "reportStyle": "detailed",
    "groupBy": "severity",
    "lastPatrolAt": "2026-05-12T17:00:00Z",
    "nextPatrolAt": "2026-05-12T18:00:00Z"
  }
}
```

### 2. 保存巡检配置
```
POST /api/v1/alert/patrol
```

**请求体：**
```json
{
  "subscriptionId": 1,
  "enabled": true,
  "patrolMode": "interval",
  "patrolInterval": 3600,
  "patrolTimes": "[]",
  "includeResolved": false,
  "timeRange": 0,
  "maxAlertsPerReport": 100,
  "sendMode": "always",
  "channelIds": "[]",
  "userIds": "[]",
  "reportStyle": "detailed",
  "groupBy": "severity"
}
```

### 3. 获取巡检报告列表
```
GET /api/v1/alert/patrol/:subscriptionId/reports?page=1&page_size=20
```

**响应示例：**
```json
{
  "code": 0,
  "data": {
    "data": [
      {
        "id": 1,
        "createdAt": "2026-05-12T18:00:00Z",
        "firingCount": 10,
        "criticalCount": 3,
        "warningCount": 5,
        "infoCount": 2,
        "sent": true,
        "sentAt": "2026-05-12T18:00:05Z"
      }
    ],
    "total": 100
  }
}
```

### 4. 手动执行巡检
```
POST /api/v1/alert/patrol/:subscriptionId/execute
```

**响应示例：**
```json
{
  "code": 0,
  "data": {
    "message": "巡检任务已启动"
  }
}
```
