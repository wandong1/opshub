# 告警通道修复说明

## 修复内容

### 1. ✅ 邮件发送使用 TLS 加密

**问题**：原来使用 `smtp.SendMail` 明文发送，不安全且容易被现代邮件服务器拒绝。

**修复**：
- 新增 `sendEmailWithTLS()` 方法：支持端口 465 的直接 TLS 连接
- 新增 `sendEmailWithSTARTTLS()` 方法：支持端口 25/587 的 STARTTLS 升级
- 根据端口自动选择加密方式：
  - 端口 465 → 直接 TLS
  - 端口 25/587 → STARTTLS

**文件**：`plugins/monitor/service/alert_service.go`

**代码位置**：第 238-424 行

---

### 2. ✅ 钉钉加签功能

**问题**：配置了 `dingtalkSecret` 但未使用，导致加签功能无效。

**修复**：
- 实现 `generateDingTalkSign()` 方法：使用 HMAC-SHA256 算法生成签名
- 在发送钉钉消息时，如果配置了密钥，自动生成签名并添加到 URL
- 签名算法：`sign = base64(hmac_sha256(timestamp + "\n" + secret))`

**文件**：`plugins/monitor/service/alert_service.go`

**代码位置**：
- `sendDingTalk()` 方法：第 494-584 行
- `generateDingTalkSign()` 方法：第 586-593 行

**使用方式**：
```go
// 如果配置了加签密钥，生成签名
if config.DingTalkSecret != "" {
    timestamp := time.Now().UnixMilli()
    sign := s.generateDingTalkSign(timestamp, config.DingTalkSecret)
    webhookURL = fmt.Sprintf("%s&timestamp=%d&sign=%s", webhookURL, timestamp, sign)
}
```

## 注意事项

1. **邮件发送**：
   - Gmail 需要使用"应用专用密码"，不能使用账号密码
   - 端口 465 使用直接 TLS，端口 587 使用 STARTTLS
   - 确保 SMTP 服务器地址和端口正确

2. **钉钉加签**：
   - 加签密钥以 `SEC` 开头
   - 签名有效期 1 小时
   - 如果不配置密钥，则不使用加签（兼容旧版）

3. **日志查看**：
   - 日志文件：`logs/app.log`
   - 使用 `tail -f logs/app.log | grep "告警"` 实时查看告警日志
   - 所有发送操作都会记录详细日志

4. **前端缓存**：
   - 如果看不到邮件选项，清除浏览器缓存
   - 开发环境：`npm run dev` 会自动热更新
   - 生产环境：需要重新 `npm run build`
