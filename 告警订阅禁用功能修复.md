# 告警订阅禁用功能修复

## 🐛 问题描述

**问题**：
- 用户无法关闭告警订阅
- 关闭订阅时报错：`{"code":400,"message":"至少需要配置一个通知通道","timestamp":1778123209}`

**影响**：
- 用户无法临时停止某些告警订阅
- 必须删除订阅才能停止推送

---

## 🔍 问题原因

### 原有逻辑

在 `subscription_handler.go` 的 `validateSubscriptionRequest` 方法中：

```go
func (s *HTTPServer) validateSubscriptionRequest(c *gin.Context, req *subscriptionRequest) error {
    // 1. 校验至少有一个通道（rule-level 或 subscription-level）
    hasChannel := len(req.ChannelIDs) > 0
    if !hasChannel {
        for _, rule := range req.Rules {
            var channelIDs []uint
            if err := json.Unmarshal(rule.ChannelIDs, &channelIDs); err == nil && len(channelIDs) > 0 {
                hasChannel = true
                break
            }
        }
    }
    if !hasChannel {
        return fmt.Errorf("至少需要配置一个通知通道")  // ❌ 导致无法关闭订阅
    }
    // ...
}
```

### 问题分析

**校验逻辑在创建和更新时都会执行**：
- ✅ 创建订阅时：必须配置通道（合理）
- ✅ 更新订阅时：如果启用，必须配置通道（合理）
- ❌ **关闭订阅时**：仍然要求配置通道（不合理）

**用户场景**：
1. 用户创建了一个告警订阅，配置了通道
2. 用户想临时关闭这个订阅（设置 `enabled: false`）
3. 前端发送更新请求，包含 `enabled: false`
4. 后端校验时仍然要求配置通道
5. 如果前端没有传递通道配置，报错"至少需要配置一个通知通道"

---

## ✅ 修复方案

### 方案：跳过已禁用订阅的通道校验

**核心思路**：
- 如果订阅的 `enabled` 字段为 `false`，跳过通道校验
- 允许用户关闭订阅而不修改通道配置
- 仍然校验数据源和标签匹配器的语法（保证数据完整性）

### 修复后的逻辑

```go
func (s *HTTPServer) validateSubscriptionRequest(c *gin.Context, req *subscriptionRequest) error {
    ctx := c.Request.Context()

    // 如果订阅已禁用，跳过通道校验（允许用户关闭订阅）
    // 但仍然校验数据源和标签匹配器的语法
    if !req.Enabled {
        // 校验数据源 ID 有效性
        for _, rule := range req.Rules {
            var dsIDs []uint
            if err := json.Unmarshal(rule.DataSourceIDs, &dsIDs); err == nil && len(dsIDs) > 0 {
                for _, dsID := range dsIDs {
                    if _, err := s.dsRepo.GetByID(ctx, dsID); err != nil {
                        return fmt.Errorf("数据源 ID %d 无效", dsID)
                    }
                }
            }
        }

        // 校验标签匹配器语法
        for _, rule := range req.Rules {
            var matchers []map[string]string
            if err := json.Unmarshal(rule.LabelMatchers, &matchers); err == nil {
                for _, m := range matchers {
                    op, ok := m["op"]
                    if !ok {
                        return fmt.Errorf("标签匹配器缺少 op 字段")
                    }
                    if op != "=" && op != "!=" && op != "=~" && op != "!~" {
                        return fmt.Errorf("标签匹配器操作符无效: %s", op)
                    }
                    if (op == "=~" || op == "!~") && m["value"] != "" {
                        if _, err := regexp.Compile(m["value"]); err != nil {
                            return fmt.Errorf("标签匹配器正则表达式无效: %s", m["value"])
                        }
                    }
                }
            }
        }

        return nil // 禁用的订阅不校验通道
    }

    // 以下是启用订阅的校验逻辑（保持不变）
    // 1. 校验至少有一个通道
    hasChannel := len(req.ChannelIDs) > 0
    if !hasChannel {
        for _, rule := range req.Rules {
            var channelIDs []uint
            if err := json.Unmarshal(rule.ChannelIDs, &channelIDs); err == nil && len(channelIDs) > 0 {
                hasChannel = true
                break
            }
        }
    }
    if !hasChannel {
        return fmt.Errorf("至少需要配置一个通知通道")
    }

    // 2. 校验通道 ID 有效性
    // 3. 校验数据源 ID 有效性
    // 4. 校验标签匹配器语法
    // ...
}
```

## 🎯 使用说明

### 1. 关闭订阅

**方式 1：通过前端界面**
- 进入"告警治理" → "告警订阅"
- 找到需要关闭的订阅
- 点击"启用"开关，关闭订阅
- 保存

**方式 2：通过 API**
```bash
curl -X PUT http://localhost:9876/api/v1/alert/subscriptions/5 \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -d '{
    "id": 5,
    "name": "测试订阅",
    "enabled": false,
    "rules": [...]
  }'
```

### 2. 重新启用订阅

**注意**：重新启用时，必须确保配置了通道

**方式 1：通过前端界面**
- 进入"告警治理" → "告警订阅"
- 找到已关闭的订阅
- 点击"编辑"，确认通道配置
- 点击"启用"开关，启用订阅
- 保存

**方式 2：通过 API**
```bash
curl -X PUT http://localhost:9876/api/v1/alert/subscriptions/5 \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -d '{
    "id": 5,
    "name": "测试订阅",
    "enabled": true,
    "rules": [
      {
        "channelIds": "[7]",  // 必须配置通道
        "userIds": "[2]"
      }
    ]
  }'
```

## 🔒 安全性说明

### 1. 禁用的订阅不会触发通知

评估引擎已经检查 `sub.Enabled`，禁用的订阅不会触发通知。

**参考代码**（`eval_service.go` 第 695-702 行）：
```go
// 检查订阅是否启用
sub, err := e.subRepo.GetByID(ctx, sr.SubscriptionID)
enabledMatched := (err == nil && sub.Enabled)
matchResult["enabled"] = enabledMatched
if !enabledMatched {
    appLogger.Debug("订阅未启用", zap.Uint("subscriptionID", sr.SubscriptionID), zap.Error(err))
    matched = false
    e.saveSubscriptionLog(ctx, sr.SubscriptionID, event.ID, rule.ID, matched, matchResult, denoiseResult, notifyResult)
    continue
}
```

### 2. 数据完整性

**禁用的订阅**：
- ✅ 可以没有配置通道
- ✅ 仍然校验数据源 ID 的有效性
- ✅ 仍然校验标签匹配器的语法

**启用的订阅**：
- ✅ 必须配置通道
- ✅ 必须校验通道 ID 的有效性
- ✅ 必须校验数据源 ID 的有效性
- ✅ 必须校验标签匹配器的语法
